Remote configuration for Android apps

Update app behavior without shipping a new version.

Remote Kit is a lightweight Remote Config platform for Android applications. It includes an Android SDK, a web admin portal, a Node.js server, MongoDB storage, targeting rules, A/B testing insights, analytics, audit logs and optimized server architecture.

Product capabilities

Feature set

Remote Kit allows developers and product teams to remotely manage UI values, feature flags, country-based targeting and A/B testing insights from a central dashboard.

Remote Config

Manage dynamic configuration keys with String, Number, Boolean and JSON values.

Automatic Android Apply

The SDK scans the screen and applies remote values to matching Android view IDs.

A/B Testing Insights

Split users into variants, collect clicks and show the leading variant in the dashboard.

Leading Variant Indicator

The dashboard highlights the variant with the highest click count using a crown icon.

Country Targeting

Return specific values for selected countries while keeping a global default value.

Scheduled Rollout

Schedule values to become active at a future date and time.

Analytics Dashboard

Show conversions, click trend over time, activity heatmap and live device data.

Audit Logs

Track changes and restore previous values using rollback.

Optimized Backend

Use in-memory cache, bulk writes, daily counters and MongoDB indexes.

Important: The dashboard can show which A/B variant is leading, but this project does not automatically apply the winner by button click. Applying a winner is listed as a possible future improvement.

Implementation

Android SDK initialization

RemoteConfig.init(
    context = applicationContext,
    apiKey = "demo-project",
    baseUrl = "http://192.168.1.115:3001",
    country = "IL",
    listener = object : RemoteConfig.Listener {
        override fun onReady() {
            runOnUiThread {
                RemoteConfig.autoApply(rootLayout)
                RemoteConfig.autoRegisterViews(rootLayout)
            }
        }

        override fun onError(message: String) {
            Log.e("RemoteKit", message)
        }
    }
)

Recommended refresh throttle

private var lastRefreshTime = 0L
private val refreshCooldownMs = 30_000L

override fun onResume() {
    super.onResume()

    val now = System.currentTimeMillis()
    if (now - lastRefreshTime < refreshCooldownMs) return

    lastRefreshTime = now
    RemoteConfig.refresh(listener = listener)
}

Recommended auto-register usage

if (BuildConfig.DEBUG) {
    RemoteConfig.autoRegisterViews(rootLayout)
}

Config lifecycle

Performance

Server efficiency

The server is optimized for frequent SDK reads and dashboard analytics. The goal is to avoid unnecessary MongoDB reads and reduce write pressure.

In-Memory Cache

After the first project request, configs are served from memory.

Project Invalidation

Only the changed project cache is cleared after publish, delete or rollback.

Bulk Device Writes

Device sync updates are buffered and flushed with MongoDB bulkWrite.

Daily Counters

Clicks are stored as daily counters instead of raw event documents.

Compound Indexes

Indexes match dashboard queries such as projectId + timestamp and projectId + lastSync.

Lean Reads

Read-only MongoDB queries can return lightweight plain JavaScript objects.

Defense explanation

Efficiency summary: SDK config reads are served from cache after the first request. Device sync updates are buffered and saved using bulkWrite. Analytics are stored as counters instead of raw events. MongoDB indexes were added according to the dashboard query patterns.

Roadmap

Future improvements

  • Add admin authentication with JWT.
  • Separate public SDK API key from private admin permissions.
  • Move server code into routes, services and models.
  • Add rate limiting for SDK endpoints.
  • Add Redis for distributed cache if multiple server instances are used.
  • Add one aggregated dashboard endpoint: GET /api/dashboard/:projectId.
  • Run autoRegisterViews only in debug mode or when the UI hash changes.
  • Add local SDK cache for offline support.
  • Add an optional manual "Apply leading variant" action for A/B tests.
